This reference explains how outbound webhooks work for your workspace: how a subscription is set up, which events can fire one, how deliveries are signed and retried, and what causes a subscription to pause automatically.

An outbound webhook lets your workspace receive an HTTPS `POST` to a URL you control whenever a chosen business event happens. Each subscription targets one event type and one destination URL. Every delivery is signed so you can verify it came from us, failures are retried, and a subscription that keeps failing is paused automatically.

The user-facing roles referenced below are **Admin**, **Ops Manager**, **Sales Admin**, **Sales Member**, **Client Coordinator**, **Crew Leader**, and **Crew Member**.

## How a webhook is configured

A webhook is a **subscription**: one record pairs a single event type with a single destination URL, and it belongs to one workspace.

<Card title="A subscription has these settings" icon="sliders">
- **Workspace** — the workspace the subscription belongs to. Set when the subscription is created and not changeable afterward.
- **Event type** — one value chosen from the fixed event list below.
- **Destination URL** — the HTTPS endpoint that receives the delivery.
- **Signing secret** — generated automatically when the subscription is created. It is shown once at creation and is never displayed, read, or edited again. Store it securely when you create the subscription.
- **Status** — `active` (the default), `paused`, or `deleted`.
</Card>

### Who can manage webhooks

<Card title="Webhook management is Admin-only and workspace-scoped" icon="shield-check">
Viewing, creating, updating, and deleting webhook subscriptions are all limited to **Admin**. Admins only see and manage subscriptions belonging to their own workspace; subscriptions never cross the workspace boundary.
</Card>

For how the workspace boundary works in general, see [Org branch scoping](/reference/platform/org-branch-scoping). For the full role breakdown, see the [Permissions roles matrix](/reference/platform/permissions-roles-matrix).

### The destination URL is validated when you save

The destination URL is rejected with a message at save time unless it passes every check:

<Card title="Destination URL rules" icon="link">
- **A URL is required.** A blank value returns "Target URL is required."
- **It must use HTTPS.** A non-HTTPS URL returns "Target URL must use HTTPS."
- **It must use a hostname, not a raw IP address.** An IP literal returns "Target URL must use a hostname, not an IP address," because an IP can never clear the delivery-time safety check and so would always fail.
- **It must not point to a private or reserved hostname** (such as a loopback name or an internal-network name). Those return "Target URL must not point to a private or reserved hostname."
- **It must be a valid URL.** A malformed value returns "Invalid URL format."
</Card>

## Events a webhook can fire on

The set of event types is a fixed list. A subscription must target one of these exact values; anything outside the list is rejected when you save.

<Card title="Supported events" icon="bolt">
- **Leads** — `lead.created`, `lead.updated`, `lead.deleted`, `lead.status_changed`
- **Jobs** — `job.created`, `job.updated`, `job.completed`
- **Work orders** — `workOrder.created`, `workOrder.updated`, `workOrder.approved`
- **Invoices** — `invoice.created`, `invoice.sent`, `invoice.paid`
- **Payments** — `payment.received`
- **Proposals** — `proposal.created`, `proposal.accepted`, `proposal.rejected`
- **Customers** — `customer.created`, `customer.updated`
- **Bug reports** — `bugReport.created`
</Card>

## Delivery behavior

When a subscribed event happens in your workspace, every **active** subscription matching that workspace and event type receives its own delivery. Each subscription's deliveries, retries, and failures are tracked separately, so a problem with one endpoint never affects another.

<Card title="What a delivery looks like" icon="paper-plane">
- It is an HTTPS `POST` to the subscription's destination URL.
- The body is JSON with two top-level keys: `type` (the event type) and `data` (the event details).
- It carries a `Content-Type` of `application/json` plus the signature headers described below.
- Redirects are not followed.
- The request times out after **30 seconds**.
- Your endpoint's response body is read but only the first portion is kept for the delivery log; anything larger is truncated.
</Card>

### Endpoint safety check at delivery time

Just before each request is sent, the destination is re-checked for safety. A request is blocked if the hostname resolves to a private, loopback, or link-local address, or to a reserved internal name. This protects against a hostname that passed validation at save time but later points somewhere it should not. A blocked request is treated as a failed attempt and retried.

### How each outcome is handled

<Card title="Delivery outcomes" icon="list-check">
- **A 2xx response** is recorded as a success.
- **A 410 Gone response** is treated as a permanently decommissioned endpoint: the attempt is recorded as failed, the subscription is paused, and it is not retried.
- **Any other non-2xx response** is recorded as failed and retried.
- **A network error, timeout, or blocked request** is recorded as failed and retried.
</Card>

## Verifying a delivery is genuine

Every delivery is signed with HMAC-SHA256 so your endpoint can confirm it came from us and was not tampered with or replayed.

<Card title="Signature headers on every delivery" icon="signature">
- `webhook-id` — a unique identifier for this delivery.
- `webhook-timestamp` — the send time in Unix seconds.
- `webhook-signature` — the signature, computed over the delivery id, timestamp, and body together, using your subscription's signing secret as the key.
</Card>

To verify, recompute the signature from the same three parts using your stored signing secret and compare it to the header. Reject the delivery if the timestamp is more than **5 minutes** away from your current time; this is your replay protection. More than one signature value may be present, separated by spaces, so a match against any one of them is valid.

## Retries and automatic pausing

<Card title="Retries" icon="rotate">
A failed delivery is retried up to **3 times**. This applies to non-2xx responses, network errors, timeouts, and blocked requests. A 410 Gone response is never retried — it pauses the subscription immediately.
</Card>

<Card title="Concurrency and rate limits" icon="gauge">
- At most **5 deliveries** are in flight at once per subscription.
- Deliveries are capped at **50 per minute per workspace**.
</Card>

<Card title="Automatic pausing on sustained failure" icon="circle-pause">
When a delivery has exhausted its retries, the recent delivery history is checked. If a subscription has **50 or more failures in the last 24 hours and no successes** in that window, it is paused automatically. Automatic pausing only moves an active subscription to paused; it never overrides a change an Admin makes at the same time, such as deleting the subscription.
</Card>

A paused subscription stops receiving deliveries until an Admin sets it back to active.

## Delivery logs

Every attempt, whether it succeeds or fails, is recorded in a delivery log for your workspace. Each entry captures the delivery id, event type, status, attempt number, the response status, a truncated copy of the response body, how long the request took, and any error message.

<Card title="Delivery log behavior" icon="clipboard-list">
- Logs are read-only and scoped to your workspace.
- The 24-hour failure and success counts that drive automatic pausing are computed from these logs.
</Card>

## Quick reference

- **Manage webhooks (view, create, update, delete):** Admin only, within their own workspace.
- **Destination URLs** must be HTTPS, must use a hostname (not an IP), and must not point to a private or reserved host.
- **Deliveries** are signed with HMAC-SHA256 and include a delivery id, timestamp, and signature header.
- **Verify** by recomputing the signature and rejecting timestamps more than 5 minutes old.
- **Retries:** up to 3 per failure; a 410 Gone pauses the subscription with no retry.
- **Limits:** 5 in-flight deliveries per subscription, 50 deliveries per minute per workspace.
- **Auto-pause:** 50+ failures and zero successes in 24 hours pauses the subscription.
